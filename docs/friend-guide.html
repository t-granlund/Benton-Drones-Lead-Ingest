What this is

The Benton Drones Lead Ingest system replaces Google Forms + PDFfiller + Google Sheets + manual Google Earth planning with one self-owned, Shopify-aware pipeline.

  • Visitors fill out a branded signup form with an electronic waiver and typed-name signature.
  • Leads, consent records, and signatures are stored in a real database (Neon PostgreSQL in production; SQLite locally).
  • Admins log in to a dashboard with analytics, a Leaflet map, breakdowns, and per-lead drill-downs.
  • Exports come out as CSV, GeoJSON, or KML for planning and reporting.
  • Optional JIRA integration creates a ticket per signup.

There are three ways to interact with it:

  1. Use the hosted app — the easiest path. Open the Render URL, submit signups, and log into the admin dashboard.
  2. Clone and run locally — great for development, testing, or playing with the code on your own Mac.
  3. Read the docs — this page, the training guide, and the architecture diagram explain how everything fits together.

End-to-end on the hosted app

  1. Open the signup form in a browser: https://benton-drones-lead-ingest.onrender.com/signup.
  2. Fill in name, email, phone, and address.
  3. Read the waiver, check the consent box, and type your full legal name as a signature.
  4. Submit. You will see a confirmation page.
  5. Go to /admin-login and enter the admin password (ask Tyler for it).
  6. Open /admin to see the new lead on the map and in the recent-leads table.
  7. Click View on a lead to see details, consent, and signature.
  8. Click CSV, GeoJSON, or KML to export the lead data.

Note Free Render tier

The free Render plan spins the web service down after ~15 minutes of inactivity. The first request after that will take roughly 50 seconds to wake up. Upgrading to Render's Starter plan removes spin-down.

2. Clone to your Mac and run locally

This is a pure Python project. No Node, no npm, no Docker required — just Python 3.11+.

Prerequisites

  • macOS with Terminal (or iTerm).
  • Python 3.11 or newer. Check with python3 --version.
  • Git. Check with git --version.
  • (Optional) pip for installing fpdf2 if you want true PDF export.

Step 1: clone the repo

Terminal
git clone https://github.com/t-granlund/Benton-Drones-Lead-Ingest.git
cd Benton-Drones-Lead-Ingest

Step 2: (optional) install dependencies

The core app uses only the Python standard library. Running the tests with make test also works without installing anything extra. If you want true PDF downloads instead of the print view fallback, install the optional requirements:

Terminal
pip3 install -r requirements.txt

Step 3: initialize the database

Terminal
python3 scripts/init_db.py

This creates data/lead_ingest.sqlite3 in the project folder.

Step 4: start the server

Terminal
ADMIN_PASSWORD=choose-a-strong-password python3 -m lead_ingest.server

You should see output like Serving on http://127.0.0.1:8000.

Step 5: open in a browser

Step 6: test the full flow locally

  1. Submit a test signup at /signup.
  2. Log in at /admin-login with the password you set.
  3. Go to /admin and confirm the lead appears.
  4. Click View, then Print / PDF to see the consent form.
  5. Download CSV, GeoJSON, or KML exports from the dashboard.

Run the test suite

Terminal
python3 -m unittest discover -s tests

# Or use the Makefile:
make test

The suite has 281 unit/integration tests plus 55 E2E tests. All should pass.

  1. A visitor clicks a link or a Shopify CTA and lands on /signup.
  2. The form captures name, contact info, service interest, consent, and a typed-name signature.
  3. The Render backend validates the input, checks CSRF and honeypot, geocodes the address, and writes to Neon PostgreSQL.
  4. If JIRA is configured, a ticket is created; otherwise the signup is queued locally.
  5. An admin logs in and reviews leads on the dashboard, map, and detail pages.
  6. Admins export data as CSV, GeoJSON, or KML for planning and reporting.

Render + Neon: what does what

The public app has three layers. It helps to know which one does what, especially when something breaks.

LayerHostWhat it does
Static guides & landing pageGitHub PagesThis guide, the architecture diagram, the training guide, and the marketing landing page. No server-side code.
Python appRenderSignup form, admin dashboard, exports, JIRA hooks, health checks. Runs as a Docker container.
DatabaseNeonPostgreSQL that stores leads, consent, signatures, variants, and JIRA queue. Persists even if Render restarts.

Why Neon?

Render's free web service has an ephemeral filesystem. If we used SQLite on Render, the database would disappear every time the container restarted or moved. Neon PostgreSQL lives outside the container, so the data persists. The app automatically switches from SQLite to PostgreSQL when the DATABASE_URL environment variable is set.

Environment variables on Render

These are already set in the live Render service. You do not need to touch them to use the hosted app, but this is what makes it work:

VariablePurpose
ADMIN_PASSWORDPassword for /admin-login.
ADMIN_SESSION_SECRETRandom 32+ char string that signs admin session cookies.
CSRF_SECRETRandom 32+ char string that signs CSRF tokens.
COOKIE_SECURESet to 1 because Render serves HTTPS.
ENVSet to production to enforce strict secret checks.
DATABASE_URLNeon connection string with ?sslmode=require.

Important notes

Legal Waiver text is a placeholder

The waiver currently shown in the signup form is clearly marked as a placeholder. Before accepting real customer signups, replace the text in lead_ingest/models.py with language reviewed by a qualified attorney.

Security Do not put secrets in the repo

The DATABASE_URL, admin password, and signing secrets live only in the Render dashboard environment variables. They are never committed to GitHub. If you run locally, set them in your terminal session, not in the code.

Good to know GitHub Pages is public

The repo and the GitHub Pages site are public. Do not paste PII, passwords, or database connection strings into any file in the repo.
