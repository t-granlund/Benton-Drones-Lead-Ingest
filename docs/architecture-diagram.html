Lead Ingest System Architecture

Self-owned, Shopify-aware lead capture: branded landing page, consent & typed-name signature, JIRA tickets, PDF export, and an admin dashboard with maps and exports. SQLite runs locally; the hosted app on Render uses Neon PostgreSQL.

Components in detail

Each node of the diagram, explained in prose.

Actor

Visitor

A prospective customer. Today they arrive via a direct link to the owned backend; tomorrow a Shopify storefront CTA will route them here. No PII is collected until the form is submitted.

Future / deferred

Shopify Storefront

Shopify runs the public storefront. The launch path uses a CTA button linking into /signup. A deeper App Proxy integration (HMAC-signed context tokens) is implemented as utilities but deferred for production.

Landing Page

A flat, branded static hero — olive gradient, Jost type, benefit cards, and a CTA into signup. Served as a static file, no server-side rendering needed.

/landing-page.html
Implemented

Signup Form

Variant-aware form capturing lead info, consent checkbox, waiver box, and typed-name signature. Hidden Shopify context, CSRF token, and honeypot field. Variants (title, subtitle, CTA, campaign, source) live in the landing_page_variants table.

/signup · /signup/{variant}
Implemented

Owned Backend

Python stdlib ThreadingHTTPServer handler. Validates input, checks CSRF tokens, trips the honeypot, rate-limits by IP+path (20/min), geocodes the address (swappable mock), and persists the signup with linked consent + signature rows. SQLite locally; switches to Neon PostgreSQL automatically when DATABASE_URL is set (Render deployment).

lead_ingest/server.py
Implemented

SQLite Database

Local: single file at data/lead_ingest.sqlite3. Hosted: Neon PostgreSQL on Render. Tables: signups, consent_records, signatures, landing_page_variants, clusters + cluster_members, jira_queue, jira_tickets. Foreign keys on; row factory enabled.

lead_ingest/db.py
Implemented

JIRA Integration

After each signup, the backend builds a JIRA Cloud REST payload (stdlib urllib only — no SDK) and creates a ticket. If JIRA env vars are unset or the call fails, the signup is queued in jira_queue for later replay.

lead_ingest/jira.py
Implemented

PDF Export

/print returns a browser-printable HTML consent form with no dependencies. /pdf returns a true PDF when fpdf2 is installed, otherwise it gracefully redirects to the print view.

/admin/lead/{id}/print · /pdf
Implemented

Admin Dashboard

Password-gated dashboard: analytics cards (total / today / week / pending geocodes), a Leaflet map of geocoded leads (vendored locally), source / campaign / variant breakdowns, a recent-lead table, and per-lead drill-down.

/admin · /admin/lead/{id}
Implemented

Admin-only CSV (with signature & consent metadata), GeoJSON (geocoded leads for the map), and KML (for Google Earth planning). These replace the old manual Google Sheets + Earth workflow.

Data flow

End-to-end path of a single lead, from arrival to admin action.

  1. A visitor clicks a Shopify storefront CTA (or a direct link) and lands on the branded Landing Page.
  2. The CTA points to /signup or /signup/{variant}, rendering the variant-aware form with a CSRF token and Shopify context (if present).
  3. The visitor submits lead info, ticks consent + waiver, and types their full legal name as a signature.
  4. The Owned Backend validates the payload, checks the CSRF token + honeypot, applies rate limiting, and geocodes the address.
  5. Signup, consent record, and signature row are persisted in SQLite — foreign-keyed and auditable.
  6. The backend attempts to create a JIRA ticket; on success it records the ticket key, on failure it queues the signup locally.
  7. The admin logs in at /admin-login and reviews analytics, the Leaflet map, breakdowns, and per-lead drill-downs.
  8. From a lead's detail page the admin prints/PDFs the consent form, or exports the full lead set as CSV, GeoJSON, or KML.

Production path

The app is deployed on Render at https://benton-drones-lead-ingest.onrender.com and uses Neon PostgreSQL for persistence. The production target is still leads.bentondrones.com behind Cloudflare. Remaining blockers:

  1. Custom domain — add leads.bentondrones.com in Render and create the DNS record in Cloudflare.
  2. Cloudflare API token scoped to bentondrones.com (zone + DNS read) so preflight scripts can verify the cutover.
  3. Preserve Google Workspace MX/SPF/DKIM/DMARC records.
  4. Smoke-test signup, admin login, exports, and JIRA against the real domain.
