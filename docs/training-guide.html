1.Public signup flow

The public-facing pages are styled to match the Benton Drones website. There are two entry points:

  • Landing page: /landing-page.html - a standalone branded page with a CTA to the signup form.
  • Signup form: /signup or /signup/<variant> - the actual lead capture form.

Public signup flow: visitor to a persisted lead with consent and signature.

What the visitor sees

  1. Visitor arrives from a Shopify CTA, ad, or direct link.
  2. Visitor reads the aerial services waiver in the scrollable waiver box.
  3. Visitor checks "I have read and agree to the above waiver".
  4. Visitor types their full legal name in the signature field.
  5. Visitor submits the form and sees a confirmation message.

Important Waiver text is a placeholder

The waiver currently shown in the form is clearly marked as a placeholder. Before accepting real signups, replace the text in lead_ingest/models.py with language reviewed by legal counsel.

Variant tracking

You can track marketing attribution by using URLs like:

/signup/facebook-ad?source=facebook&campaign=summer2026

The form automatically captures source, campaign, and variant_slug. These values appear in the admin dashboard and exports.

3.Consent & signature audit trail

Every signup captures a verifiable audit trail. The consent and signature records are stored in separate tables linked to the signup by a foreign key, so the exact text a user agreed to is preserved forever, even if you change the waiver later.

Consent version + text snapshot

The exact consent version (2026-07-06.v1) and the full consent text are copied into the consent_records row at the moment of signup. Changing the text later does not alter past records.

Accepted timestamp

An ISO 8601 UTC timestamp is written to accepted_at the instant the form is submitted and the row is created.

IP address + user agent

The visitor's IP address and browser user-agent string are captured for both the consent record and the signature record, supporting dispute resolution.

Typed full legal name

The visitor types their full legal name into the signature field. The typed value is stored verbatim in full_name_typed in the signatures table.

Waiver version + text snapshot

The waiver version (2026-07-15.placeholder.v1) and the complete waiver text are snapshotted into the signature row, so you always know which wording was agreed to.

Signature disclaimer text

The disclaimer stating that typing a name constitutes a legally binding electronic signature is itself stored alongside the signature, so the full context of agreement is preserved.

Why two separate tables?

Consent (the permission to be contacted) and signature (the waiver agreement) are legally distinct acts. Storing them in separate, foreign-keyed tables means each can be audited independently and survives future changes to either document without retroactively rewriting history.

4.Exports & PDF

CSV export

The CSV export at /export/csv includes lead data plus consent and signature metadata (signed name, signed at, waiver version). Use it for spreadsheet reporting and planning.

GeoJSON / KML

Download /export/geojson or /export/kml to view leads in mapping tools like Google Earth or QGIS.

PDF of completed form

Each lead has two export options:

  • /admin/lead/<id>/print - browser-printable HTML view. No dependencies. Use Ctrl+P / Cmd+P to save as PDF.
  • /admin/lead/<id>/pdf - true PDF download if fpdf2 is installed. Otherwise it falls back to the print view.
Install optional PDF dependency 
pip install fpdf2

5.JIRA integration

The system can create a JIRA ticket for every completed signup. Set these environment variables:

JIRA environment variables 
JIRA_BASE_URL=https://your-domain.atlassian.net
JIRA_USER_EMAIL=you@example.com
JIRA_API_TOKEN=your-api-token
JIRA_PROJECT_KEY=BDS
JIRA_ISSUE_TYPE=Task  # optional, defaults to Task

JIRA flow: if configured, a ticket is created; otherwise the signup is queued for later replay.

If JIRA is not configured

Signups are stored in a local jira_queue table with status pending. You can process them later by:

  1. Setting the JIRA env vars.
  2. Running a future processing script (not yet built - add if needed).

Privacy note

JIRA ticket descriptions include the lead name, email, address, and signature metadata. Make sure your JIRA project permissions are appropriate for PII.

7.Deployment

Local vs hosted vs deployed

Static documentation pages (this guide, the architecture diagram, the explainer guide, and the landing page) are hosted on GitHub Pages and work without any server:

https://t-granlund.github.io/Benton-Drones-Lead-Ingest/

The interactive app (signup form, admin dashboard, exports, maps) requires the Python backend. You can either:

  1. Run it locally — git clone, make init-db, ADMIN_PASSWORD=... make run. Best for development and testing.
  2. Deploy it to Railway — The repo includes a Dockerfile and railway.json. Railway gives you a public HTTPS URL with persistent SQLite storage. See docs/railway-deployment-guide.md for the full walkthrough.
  3. Deploy it to Fly.io — An alternative host with a fly.toml example in the deployment guide.

GitHub Pages cannot run Python or SQLite, so the dynamic routes (/signup, /admin, /export/*) only work when the app is running locally or on a hosted platform like Railway or Fly.io.

Database: The app uses SQLite by default (zero config, single file). For production with multiple instances, set the DATABASE_URL environment variable to a PostgreSQL connection string (e.g. Neon) and the app switches automatically — no code changes needed. See docs/railway-deployment-guide.md for the Neon setup walkthrough.

Deployment flow: local dev to tests to reverse proxy to HTTPS production.

Reverse proxy setup

Example configs are provided in deploy/Caddyfile and deploy/nginx.conf. In production, always:

  1. Run the Python app on 127.0.0.1:8000.
  2. Put Caddy or nginx in front for TLS termination.
  3. Forward the X-Forwarded-Proto: https header so cookies get the Secure flag.
  4. Set COOKIE_SECURE=1 or rely on the forwarded header.
  5. Keep ADMIN_PASSWORD and secrets in a vault or environment, never in code.

Health check

Load balancers can use /healthz for health checks. It returns JSON:

{"status": "ok", "version": "0.2.0", "tests_passed": true}

8.What gets stored (data model)

All data lives in a single SQLite file at data/lead_ingest.sqlite3 (or PostgreSQL when DATABASE_URL is set). The three tables below form the auditable core; the rest support variants, clustering, and JIRA.

signups

  • id primary key
  • name, email, phone
  • address + geocode
  • source, campaign, variant
  • shopify context
  • created_at, updated_at

consent_records

  • signup_id FK, unique
  • consent_version
  • consent_text (snapshot)
  • accepted, accepted_at
  • ip_address, user_agent

signatures

  • signup_id FK, unique
  • full_name_typed
  • waiver_version
  • waiver_text (snapshot)
  • signature_disclaimer_text
  • ip_address, user_agent, signed_at

landing_page_variants

  • slug (unique)
  • title, subtitle, cta_text
  • campaign, source
  • is_active

jira_queue

  • signup_id FK
  • ticket_key, status
  • attempted_at, error_message

jira_tickets

  • signup_id FK, unique
  • ticket_key
  • jira_issue_url

clusters

  • name, description
  • center lat / lng
  • radius_miles

cluster_members

  • cluster_id FK
  • signup_id FK
  • distance_from_center_miles

Foreign keys are enabled on every connection. The consent and signature rows are created in the same transaction as the signup, so a lead is never partially stored.

9.Security checklist

  • Replace the placeholder waiver text with legally reviewed language.
  • Use strong, unique ADMIN_PASSWORD, ADMIN_SESSION_SECRET, and CSRF_SECRET.
  • Run behind HTTPS with COOKIE_SECURE=1.
  • Restrict admin access to trusted networks if possible (at the proxy level).
  • Back up the SQLite database regularly (e.g., data/lead_ingest.sqlite3).
  • Review JIRA project permissions because ticket descriptions contain PII.
  • Keep the app and Python runtime patched.

11.Frequently asked questions

Can I run this without internet?

Yes, mostly. The app, database, admin dashboard, map tiles (vendored locally), and all tests run entirely offline. The only features that need the network are JIRA ticket creation (calls the Atlassian REST API) and the optional real geocoder (if you swap in one that calls an external API). The default MockGeocoder works offline.

Where is data stored?

All data is in a single SQLite file at data/lead_ingest.sqlite3. There is no external database server, no cloud database, and no SaaS dependency. You can copy, back up, or inspect the file with any SQLite tool.

Can I use a real geocoder instead of the mock?

Yes. The geocoder is swappable. Replace MockGeocoder in lead_ingest/geocoding.py with a class that calls a real geocoding API (such as Nominatim, Google, or Mapbox) and returns the same (latitude, longitude, display_name, provider) result. The rest of the system picks it up automatically.

Is the waiver legally valid?

The current waiver text is a clearly labeled placeholder. It captures a full audit trail (typed name, timestamp, IP, user agent, versioned text snapshot), but the wording itself must be reviewed by legal counsel before you accept real signups. The audit trail mechanism is sound; the words are not.

Can I integrate with Shopify?

Yes. See docs/shopify-integration-plan.md for the full plan. Today the launch path is a Shopify storefront CTA button linking to /signup. A deeper App Proxy integration (HMAC-signed context tokens) is implemented as utilities in lead_ingest/shopify_security.py and deferred for production.

How do I back up data?

Copy the SQLite file. The database is a single file at data/lead_ingest.sqlite3. Because WAL mode is enabled, you may also see -wal and -shm files; copy all three together for a consistent backup, or use the .backup command from the SQLite CLI for a hot backup.

What Python version is required?

Python 3.11 or later. The code uses modern typing syntax (union pipes, from __future__ import annotations) and standard library modules only for the core app. The optional PDF export needs fpdf2 from PyPI.

How do I reset the database?

Delete the database file and re-run make init-db. The file is at data/lead_ingest.sqlite3. Removing it (and any -wal / -shm sidecar files) wipes all leads, consent records, and signatures. Then make init-db recreates a fresh schema with the default landing page variant.

12.Before you share with a friend

A quick checklist to run through before handing the repo to someone else. Each item keeps the system safe and runnable on a fresh machine.

  • Verify no secrets are committed. Check .gitignore excludes .env, data/, and logs. Never hard-code passwords or API tokens.
  • Run the full test suite. make test should print 281 tests and OK.
  • Confirm the waiver is a placeholder. Make sure your friend knows the waiver text needs legal review before real use.
  • Test the clone-and-run path. git clone, make init-db, ADMIN_PASSWORD=... make run should work from a clean checkout.
  • Smoke-test the flows. Open /signup, submit a test lead, log in at /admin-login, view the lead, and try an export.
  • Point them at the docs. This training guide, the architecture diagram, and the explainer guide cover the whole system.

Remaining blockers before production

  • Replace placeholder waiver text with legally reviewed language.
  • Provision a host and DNS record for leads.bentondrones.com.
